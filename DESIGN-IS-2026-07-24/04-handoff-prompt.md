Redesign principles in priority order:
1. #6 Honest — every label and claim should map 1:1 to what actually happens or is actually true; no CTA should promise an outcome the destination doesn't deliver, no stat or superlative should stand without a real source, no content should present as current when it's stale.
2. #3 Aesthetic — every text/background pairing that traces to the design-token system should actually clear WCAG AA in practice, not just in the token's own isolated math; this means auditing every CSS rule that applies a color token to a light-surface context, not trusting that fixing one named instance fixed the underlying pattern.
3. #4 Understandable — a first-time visitor should be able to predict what any CTA does before clicking it, especially the ones repeated most often across the site.

Deliverables for the plan:
- Per-fix: target files, exact change (CSS selector correction, copy rewrite, or CTA re-labeling), and a verification step that live-checks the fix the same way this audit did (computed styles, real contrast math, not eyeballing) — this project has an established, working pattern for this (gstack `browse`, per its `CLAUDE.md`).
- A single consolidated pass over every `color: var(--accent)` / `background: var(--accent)` usage in `css/base.css` to confirm no other light-surface instance of this bug class exists beyond the three named here — don't assume the list above is exhaustive without re-checking.
- A copy-honesty pass: for each flagged claim, either a real source/number from the site owner or a softened rewrite; for the CTA labels, a single consistent decision (rename to match reality, e.g. "Enquire Now"/"Join Waiting List," applied everywhere "Book Now" currently appears) rather than fixing instances inconsistently.
- Explicit flag to the site owner: the School Holiday Camps dates need real input (current/upcoming dates or removal) — this is a content dependency the plan should surface, not silently work around.
- States checklist re-confirmation for anything touched: empty/loading/error/success/focus/disabled, matching this project's existing QA bar.
- A follow-up `/claude-mem:design-is` re-audit recommendation after this plan lands, to confirm the score actually moves — don't just assume it worked.

Anti-patterns to guard against (specific to this redesign):
- Patching the exact three named instances of the `--accent`-on-light bug without grepping for a fourth, fifth, or sixth — the whole point of move #1 is fixing the bug class, not the bug reports.
- Renaming CTA labels without checking the trust chain nav item ("Waiting List") already sets a plainer, more honest expectation — the fix might be closer to a single terminology sweep than N independent edits.
- Reaching for `!important` or duplicating the `.eyebrow`/`::before` rule in a way that re-introduces the original ~1.56:1 problem instead of the correct `gold-700` value — re-derive the fix from the actual selector/property separation, not a quick patch over today's broken one.
- Treating the stale-dates finding as something this plan can resolve without the site owner's input — it cannot invent real dates.
- Re-litigating the token system, component library, motion system, or IA — all confirmed strong this audit; scope creep into those areas is out of bounds for this plan.
